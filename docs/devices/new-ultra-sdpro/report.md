## Compatibility with this integration

**Summary: Almost nothing works out of the box.** This device runs a completely different firmware from the stock GeekMagic firmware the integration was built for. Every major API path the integration calls returns a 404, and the device's own API surface (`GET /api/set?key=&value=` for writes, `POST /upload`, `GET /restart`) has no overlap with the integration's expected endpoints. The integration will fail at the connection test step and never reach the render/upload cycle.

### What should work

Nothing aligns. Not a single endpoint the integration calls (`/app.json`, `/space.json`, `/brt.json`, `/set?brt=`, `/set?theme=`, `/set?img=`, `/doUpload?dir=/image/`, `/delete?file=`, `/set?clear=image`, `/set?page=`, `/set?enter=`, `/set?reboot=`) was found on this device. The `upload_and_display()` core loop — render image → upload → display — cannot function without all three steps working.

### What likely breaks

**`test_connection()` / integration setup fails immediately.**
`test_connection()` calls `get_space()` which fetches `GET /space.json` (`device.py:124`). This device has no `/space.json` endpoint — it was not discovered in any crawl pass. The integration will raise `aiohttp.ClientResponseError` (404) and return `ConnectionResult(success=False, error="http_error")`. The config entry setup will refuse to complete. (`device.py:281-329`)

**`detect_model()` returns `MODEL_UNKNOWN` and silently misroutes uploads.**
`detect_model()` tries `GET /.sys/app.json` (Pro path) then `GET /app.json` (Ultra path) (`device.py:371-404`). Neither exists on this firmware. Model stays `MODEL_UNKNOWN`. Downstream, `set_theme_custom()` calls `set_theme(3)` for non-Pro models (`device.py:185`) — but `GET /set?theme=N` does not exist on this device; the config write key is `theme` sent via `GET /api/set?key=theme&value=N`.

**`get_state()` — endpoint missing, 404.**
`get_state()` fetches `GET /app.json` and reads `data.get("theme", 0)`, `data.get("brt")`, `data.get("img")` (`device.py:99-114`). This device has no `/app.json`. The state poll throws a 404 on every coordinator cycle, logged as a device-offline condition and triggering exponential backoff. Even if the endpoint existed, the device stores these fields in `GET /config` under the keys `theme`, `brightness`, and `gifnum` — there is no `brt` or `img` field.

**`get_space()` — endpoint missing, 404.**
`get_space()` fetches `GET /space.json` and reads `data.get("total", 0)`, `data.get("free", 0)` (`device.py:124-138`). This device does not expose `/space.json`. Storage info is available but fragmented: `/config` returns `freespace` (int bytes free), and `/photo/list` returns `total` + `used` bytes — neither matches the integration's expected endpoint or field names.

**`get_brightness()` — endpoint missing, 404.**
`get_brightness()` fetches `GET /brt.json` expecting `{"brt": "71"}` (`device.py:148-154`). This device has no `/brt.json`. The current brightness is in `GET /config` under the key `brightness` (integer, not string). The coordinator polls brightness every 10 minutes (`coordinator.py:1049-1059`); each poll will log a debug-level exception and leave `_device_brightness` as `None`.

**`set_brightness(N)` — wrong endpoint and wrong parameter key.**
`set_brightness()` calls `GET /set?brt={value}` (`device.py:164`). The correct SD_PRO call is `GET /api/set?key=lcd_brightness&value={value}` (the JS sends key `lcd_brightness` on the slider input event — see `attachEventListeners()` in `javascript.js`). The brightness range on this device is 2–99 (from `<input type='range' id='ledBrightness' min='2' max='99'>` in the `/` body_preview), not 0–100 as the integration clamps to (`device.py:162`).

**`set_theme(N)` / `set_theme_custom()` — wrong endpoint.**
`set_theme()` calls `GET /set?theme={N}` (`device.py:175`). The correct SD_PRO call is `GET /api/set?key=theme&value={N}`. The device's theme model is also fundamentally different: the firmware cycles through enabled built-in themes (Classic, Weather, Photo, Dial, Simple, Weather forecast, Flip Clock) — there is no "custom image" theme number analogous to theme 3 or 4 from stock firmware.

**`upload()` — wrong endpoint, wrong path structure.**
`upload()` posts to `POST /doUpload?dir=/image/` with multipart field `file` (`device.py:226-244`). This device's upload endpoint is `POST /upload` (no `dir=` query parameter). The `/doUpload` path does not exist and will 404. Additionally, the SD_PRO has two distinct upload roles: `POST /photo/upload` (field `file`, photos for the slideshow) and `POST /upload` (field `imageFile`, 80×80 GIF for the GIF slot).

**`set_image(filename)` — no equivalent concept.**
`set_image()` calls `GET /set?img=/image/{filename}` (`device.py:197`). The SD_PRO firmware has no endpoint that selects a specific image to display on demand. The photo slideshow is managed entirely by the device (enabled/disabled per file via `/photo/toggle`, rotation interval via `/photo/interval`). There is no mechanism to push and immediately display a named image as the integration's rendering pipeline requires.

**`delete_file(path)` — wrong endpoint.**
`delete_file()` calls `GET /delete?file={path}` (`device.py:270`). The SD_PRO equivalent is `GET /photo/delete?name={name}` (filename only, not a full path with `/image/` prefix).

**`clear_images()` — no equivalent.**
`clear_images()` calls `GET /set?clear=image` (`device.py:277`). No equivalent exists on this firmware.

**`navigate_next/previous/enter()` and `reboot()` — wrong endpoints.**
All Pro navigation calls use `GET /set?page=` / `GET /set?enter=` / `GET /set?reboot=1` (`device.py:334-369`). The SD_PRO restart is `GET /restart` (no query parameter). Navigation buttons have no equivalent.

**`coordinator.py` display mode sync — never reached.**
The coordinator syncs display mode by reading `device_state.theme` and treating `theme < 3` as built-in mode (`coordinator.py:1070-1077`). Since `get_state()` always fails (404), `_device_state` stays `None` and the sync block is never reached. The coordinator logs the failure at debug level and continues, but the entire render/upload loop fails immediately at `upload_and_display()`.

### Extra capabilities this device has

- **Theme rotation system** — `GET /theme/list` returns 7 named built-in themes each with an `enabled` flag; `GET /theme/toggle?id=N&state=0|1` and `GET /theme/interval?val={seconds}` control per-theme enabling and rotation interval. No integration equivalent.
- **Photo slideshow with per-file control** — `GET /photo/list` returns all uploaded photos/GIFs with `enabled`/`size`; `POST /photo/upload` accepts JPEG/PNG/GIF (auto-cropped to 240×240 on device); per-file enable/disable via `/photo/toggle`; rotation interval via `/photo/interval`.
- **80×80 GIF animation slot** — `POST /upload` (field `imageFile`) uploads a separate looping GIF that appears as one of the built-in theme modes.
- **Night mode with configurable hours and brightness** — stored in `/config` as `nightmode`, `starttime`, `stoptime`, `nightbrightness`; writable via `/api/set`.
- **OTA firmware update** — `POST /update_ota` with field `update`; firmware filename must start with `SDP`.
- **RGB color customization** — three color pickers (color1, color2, color3 as RGB565 integers in `/config`); also individual R/G/B channel writes via `ws_r`/`ws_g`/`ws_b` api/set keys.
- **Unified config API** — all settings read from one `GET /config` call and written via `GET /api/set?key=&value=`. This is a cleaner pattern than stock firmware's fragmented endpoint surface and would be straightforward to adapt if adding a new device driver.

## Endpoints by feature

### Identification & system

- `GET /config` — Read all device configuration in one call. Key fields: `theme` (int, current built-in theme index 0–6), `brightness` (int, 2–99), `nightmode` (int 0/1), `starttime`/`stoptime` (int hours), `nightbrightness` (int, 2–99), `city` (string, weather city — redacted), `celsius` (bool), `hour12` (bool), `mile` (bool), `sync` (bool), `timeformat` (int 0–2), `ntp` (string), `weatherkey` (string, redacted), `ssid`/`password` (redacted), `freespace` (int bytes free), `color1`/`color2`/`color3` (int RGB565-encoded), `gifnum` (int, active GIF slot index).
- `GET /api/set?key={key}&value={value}` — Universal config write; URL-encode value. Known write keys from JS: `lcd_brightness` (2–99), `theme` (int), `gifnum` (int), `timeformat` (int), `city` (string), `celsius` (0/1), `time12_24` (0/1), `mile` (0/1), `timesync` (0/1), `zone` (numeric UTC offset), `nightmode` (0/1), `starttime` (int hour), `stoptime` (int hour), `nightbrightness` (2–99), `ntp` (string), `weatherkey` (string), `ssid` (string), `password` (string), `color1`/`color2`/`color3` (int RGB565), `ws_r`/`ws_g`/`ws_b` (int 0–255).
- `GET /restart` — Reboot the ESP8266. No parameters. The JS notes the fetch may fail because the network drops before a response arrives — treat any error as success.
- `POST /update_ota` — OTA firmware update. **Inputs:** multipart field `update`, `.bin` file; filename must start with `SDP` (validated client-side). **Effect:** flashes firmware, device reboots.

### Display

- `GET /theme/list` — Returns `{ interval: int, themes: [{ id: int, name: string, enabled: bool }] }`. Seven built-in themes: 0=Classic, 1=Weather, 2=Photo, 3=Dial, 4=Simple, 5=Weather forecast, 6=Flip Clock. Photo (id=2) shows the photo slideshow.
- `GET /theme/toggle?id={N}&state=0|1` — Enable or disable a built-in theme from the rotation. The JS enforces at least one enabled theme client-side.
- `GET /theme/interval?val={seconds}` — Set rotation interval between enabled themes. Options from UI: 8 choices (exact values redacted; interval stored as int in `/theme/list` response).
- `GET /api/set?key=lcd_brightness&value={N}` — Set brightness. Range: 2–99 (from `<input type='range' min='2' max='99'>` on `#ledBrightness` in the `/` body_preview). Note: stored as `brightness` in `/config`; the write key is `lcd_brightness` (discrepancy worth confirming on hardware).
- `GET /api/set?key=nightmode&value=0|1` — Toggle night mode.
- `GET /api/set?key=nightbrightness&value={N}` — Set night mode brightness (2–99, same slider range as global brightness).
- `GET /api/set?key=color1&value={RGB565}`, `color2`, `color3` — Set clock display colors (RGB565-encoded 16-bit integers).
- `GET /api/set?key=ws_r&value={R}`, `ws_g`, `ws_b` — Set primary color by individual channel (0–255). Sent with 50ms delays between channels to avoid packet collision (per JS comments).

### Network / WiFi

- `GET /scanwifi` — Returns a JSON array with an `ssidList` string of HTML `<option>` elements listing nearby SSIDs. All SSID values redacted by script default.
- `GET /connect` — HTML form action for WiFi credential submission (`new_ssid`, `password` fields). In practice the JS intercepts form submit and sends each value separately via `/api/set`. Whether the server also handles direct `GET /connect?new_ssid=X&password=Y` is unconfirmed.
- `GET /api/set?key=ntp&value={server}` — Update NTP server address (e.g. `pool.ntp.org`).
- `GET /api/set?key=timesync&value=1` + `GET /api/set?key=zone&value={offset}` — Sync time from browser and set UTC offset in hours. Sent 200ms apart (per JS).

### Content & data

#### Photo slideshow

- `GET /photo/list` — Returns `{ files: [{ name: string, size: int, enabled: bool }], total: int, used: int, interval: int }`. Total and used are in bytes; interval is the current slideshow rotation in seconds.
- `POST /photo/upload` — Upload a new photo. **Inputs:** multipart field `file`; the `/photo` page's JS auto-crops JPEG/PNG to 240×240 before upload via Canvas API; GIF accepted up to 240×240 as-is. **Effect:** adds file to the photo slideshow pool.
- `GET /photo/toggle?name={filename}&state=0|1` — Enable or disable a specific photo in the slideshow rotation.
- `GET /photo/delete?name={filename}` — Remove a photo from the device storage.
- `GET /photo/interval?val={seconds}` — Set the photo slideshow rotation interval. Options from UI dropdown (exact values redacted).

#### GIF animation slot (legacy)

- `POST /upload` — Two upload roles in the JS: (1) 80×80 animated GIF with field `imageFile` for the dedicated GIF animation display mode; (2) generic root-filesystem file upload with field `file` (used to update `index.html`, `javascript.js`, `style.css` in-place). The GIF must be exactly 80×80; dimensions are validated client-side before upload.

#### Weather & clock

- `GET /api/set?key=city&value={city}` — Set the OpenWeatherMap city name for weather themes.
- `GET /api/set?key=weatherkey&value={key}` — Set the OpenWeatherMap API key.
- `GET /api/set?key=celsius&value=0|1` — Toggle temperature units (Fahrenheit=0, Celsius=1).
- `GET /api/set?key=timeformat&value={N}` — Set time display format (3 options; exact meanings unconfirmed).
- `GET /api/set?key=time12_24&value=0|1` — Toggle 12/24-hour clock format.
- `GET /api/set?key=gifnum&value={N}` — Select active built-in GIF animation slot index.

## Things to verify manually

- **`GET /api/set` response body** — The JS uses fire-and-forget (`fetch(...).catch(console.error)`) with no body parsing. Whether the server returns `{"ok": true}`, `200 OK` with empty body, or something else is unknown. A new driver would need to know this to detect errors.
- **`GET /restart` response timing** — The JS notes fetch may error because the device drops the connection before responding. Whether any response body arrives is unknown.
- **Brightness write key** — `/config` returns `brightness` (int) but the JS writes `lcd_brightness` via `/api/set`. Whether writing `key=brightness` also works is unconfirmed. Both should be tested on hardware.
- **`GET /connect` direct form submission** — The HTML `<form action="/connect">` suggests the server handles direct `GET /connect?new_ssid=X&password=Y`, but the JS always intercepts it and uses `/api/set` instead. Confirm whether the `/connect` endpoint actually works server-side.
- **`theme` write key semantics** — Writing `GET /api/set?key=theme&value=2` presumably switches immediately to theme 2 (Photo). Whether this overrides the rotation cycle or just sets a starting point is unclear.
- **`gifnum` field** — `/config` returns `gifnum: 4` and the `<select id="giflist">` dropdown has 5 options (all redacted). Whether this selects among pre-loaded built-in GIF animations stored in firmware, or among user-uploaded GIFs, is unclear.
- **`/photo/upload` vs `POST /upload`** — Whether these hit the same server handler or two different ones (and whether the multipart field name `file` vs `imageFile` matters to the server) needs hardware confirmation.
- **Night mode time unit** — `/config` shows `starttime: 23` and `stoptime: 8`. These appear to be hour-of-day integers (23:00 to 08:00), but the exact semantics (local time? UTC? 24h?) need confirmation.
