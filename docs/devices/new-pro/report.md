## Compatibility with this integration

**Summary: Partial ‚Äî the core upload/display pipeline should work, but model detection and state polling are broken.** The device uses a split API surface: display control (brightness, theme, image set, upload, delete) lives at root paths matching what the integration expects; device state and settings live under `/.sys/` paths the integration doesn't poll. The most dangerous gap is `get_state()`: `/app.json` returns a 404, so every coordinator cycle logs a silent exception and `_device_state` stays stale. Fortunately the render/upload loop succeeds independently of state polling.

### What likely breaks

**`detect_model()` returns `MODEL_UNKNOWN` ‚Äî device has no `/.sys/app.json` (404).** The integration first tries `GET /.sys/app.json` to fingerprint a Pro device (`device.py:381-389`). This device returns 404 for `/.sys/app.json` (confirmed in the "Probed but not served" section of `report.raw.md`). The fallback tries `GET /app.json` for Ultra (`device.py:393-401`), which also 404s ‚Äî this device uses `/v.json` for identification, not `/app.json`. Result: `model` stays `MODEL_UNKNOWN`. (`device.py:371-404`)

**`detect_model()` consequence ‚Äî wrong custom-image theme number.** With `MODEL_UNKNOWN`, `set_theme_custom()` calls `set_theme(3)` (Ultra's custom theme). But on this firmware, theme 3 is "4.Weather" and theme 4 is "5.Picture" (custom image), per the `settings.html` `<select id="theme">`. The integration's rendered JPEG uploads correctly to `/image/dashboard.jpg`, but the device is switched to Weather mode instead of Picture mode ‚Äî the uploaded image is never displayed. This is the most critical failure for actual usage. (`device.py:179-186`, `coordinator.py:1120`)

**`get_state()` ‚Äî endpoint missing, silent 404 every coordinator cycle.** `get_state()` fetches `GET /app.json` and reads `data.get("theme", 0)`, `data.get("brt")`, `data.get("img")` (`device.py:99-114`). This device has no `/app.json` ‚Äî listed as 404 in "Probed but not served". The coordinator catches the exception at debug level (`coordinator.py:1079-1080`), so `_device_state` stays `None`. The display mode sync block (`coordinator.py:1068-1077`) is never reached, meaning:
- The coordinator cannot detect if the device was manually switched to a built-in theme.
- `brightness` and `current_image` sensors in HA always report `None`.
- The actual brightness is available from `GET /.sys/brt.json` (successfully probed: `{"brt": "85"}`) but the integration never calls this path. (`device.py:91-114`)

**`get_brightness()` ‚Äî wrong path.** `get_brightness()` fetches `GET /brt.json` (`device.py:148`). This device exposes brightness at `GET /.sys/brt.json` (successfully probed: `{"brt": "85"}`), not at the root `/brt.json` path. Every 10-minute brightness poll logs a debug exception and leaves `_device_brightness` as `None` until overridden by a write. (`device.py:140-154`, `coordinator.py:1051-1059`)

## Things to verify manually

- **`/set?brt=N` write semantics vs. stored inversion** ‚Äî The `settings.js` JS reads back brightness as `255 - res.brt` before populating the slider, suggesting internal storage is inverted. Whether `/set?brt=50` stores 50 or 205 (i.e. whether the inversion applies to reads only or also to writes) needs hardware confirmation. The integration sends raw 0‚Äì100 values.
- **`detect_model()` fix path** ‚Äî The correct fingerprint for this firmware is `GET /v.json` returning `"m": "GeekMagic SmallTV-PRO"`. If `detect_model()` were updated to check `/v.json` first, it could reliably return `MODEL_PRO`. The existing `/.sys/` path heuristic does not work for this firmware version.
- **`/.sys/app.json` dead-code status** ‚Äî `settings.html` calls `getData('/.sys/app.json')` in the body `onload` handler even though the endpoint 404s. Whether this is dead code from a prior firmware version or a lazily-populated endpoint is unclear.
- **`wifisave` parameter format** ‚Äî The `s` query param is URL-encoded JSON per JS evidence. The exact object schema (keys: `ssid`, `password`, or `s`, `p`?) needs confirmation from `network.html` JS (page is gzip-compressed; the crawl returned a binary blob, not a body_preview).
- **Night-mode time semantics** ‚Äî `t1`/`t2` appear to be hour-of-day integers (0‚Äì23 from the `<select>` options), but whether they are local time or UTC is unconfirmed.
- **Theme 7 ("Ideas")** ‚Äî Not documented in the integration or prior device reports. Semantics unknown.
- **Crypto coin ID format** ‚Äî Whether `cin` slots (bitcoin.html / CMC) take numeric IDs and `gin` slots (coingecko.html) take CoinGecko string IDs, or whether both accept the same format, is unconfirmed.
- **`/.sys/app.json` vs. `get_state()` alternative** ‚Äî Whether `/.sys/brt.json` + a different state endpoint could be combined to replicate the `get_state()` contract (theme, brightness, current image) for this firmware needs investigation.
